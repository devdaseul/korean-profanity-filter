# Spring AI 교육 발표 스크립트
## 한국어 비속어 필터 에이전트 (Korean Profanity Filter Agent)

> **대상 청중**: Spring Boot 기초를 아는 초보 개발자  
> **핵심 메시지**: Spring AI가 제공하는 추상화 인터페이스를 활용해서, LLM·임베딩·벡터DB를 어떻게 서비스에 통합하는지 직접 구현하며 익힌다

---

## 0. 발표 전 한 줄 요약

> "욕설을 단순 키워드 비교가 아니라, 정규화 → 사전 매칭 → 벡터 유사도 → LLM 판단 순으로 걸러내는 다층 필터를 Spring AI로 만들어봤습니다."

## 1. 왜 이걸 만들었나요? (목적과 취지)

서비스를 운영하다 보면 채팅·댓글에 비속어가 들어옵니다.  
가장 흔한 대응은 **블랙리스트 단어 테이블**을 두고 `contains()`로 비교하는 방식인데, 이것만으로는 한계가 있습니다.

| 우회 유형 | 예시 | 단순 비교 탐지 |
|-----------|------|--------------|
| 특수문자 삽입 | `씨@발` | ❌ |
| 숫자 치환 | `ㅅ1ㅂ` | ❌ |
| 공백 삽입 | `시 발` | ❌ |
| 영타 입력 | `tlqkf` | ❌ |
| 신조어 | `엠창`, `찐따` | ❌ |
| 애매한 맥락 | "너 지능 처참하네" | ❌ |

이 한계를 극복하기 위해 **4단계 필터 파이프라인**을 설계했습니다.  
그리고 이 파이프라인을 구현하는 핵심 도구로 **Spring AI**를 선택했습니다.

## 2. 서비스 전체 구조 (아키텍처)

```
사용자 입력
    │
    ▼
[FilterController / ProfanityAroundAdvisor]
    │
    ▼
┌──────────────────────────────────────────────┐
│              PipelineService                  │
│                                              │
│  L1: LocalFilterService ──────────────────   │
│       Step1. TextNormalizer (전처리)          │
│       Step2. BadWordFilter.matchesRegex()     │ ─→ 탐지 시 즉시 차단
│       Step3. BadWordFilter.matchesBlacklist() │ ─→ 탐지 시 즉시 차단
│       Step4. BadWordFilter.matchesFuzzy()     │ ─→ 탐지 시 즉시 차단
│                                              │
│  L2: RagFilterService (VectorStore 유사도)    │ ─→ 탐지 시 즉시 차단
│                                              │
│  L3: LlmFilterService (Ollama LLM 최종판단)   │ ─→ 판정 + 로그 저장
└──────────────────────────────────────────────┘
    │
    ▼
[PostgreSQL + pgvector]
  ┌─────────────────────────────────────────┐
  │  bad_word          (블랙리스트 사전)      │
  │  profanity_vectors (임베딩 벡터)         │
  │  llm_judge_log     (LLM 판정 기록)       │
  └─────────────────────────────────────────┘
```

**포인트**: 각 단계에서 비속어가 탐지되면 **즉시 차단(Fast-Exit)** 합니다.  
뒤 단계로 갈수록 비용(시간·연산)이 높기 때문에, 앞단에서 최대한 걸러내는 설계입니다.

## 3. Spring AI란 무엇인가? (이 프로젝트에서 쓴 인터페이스 중심으로)

Spring AI는 LLM·임베딩·벡터스토어를 **공통 인터페이스**로 추상화한 Spring 생태계 라이브러리입니다.  
벤더(OpenAI, Ollama, Azure 등)가 바뀌어도 **코드를 수정하지 않아도 됩니다.**

이 프로젝트에서 직접 사용한 Spring AI 인터페이스/클래스를 하나씩 설명합니다.

### 3-1. `ChatClient` — LLM 호출 창구

```java
// ChatClientConfig.java
@Bean @Primary
public ChatClient chatClient(ChatClient.Builder builder,
                             ProfanityAroundAdvisor profanityAroundAdvisor) {
    return builder
        .defaultAdvisors(profanityAroundAdvisor, new SimpleLoggerAdvisor())
        .build();
}
```

- `ChatClient`는 Spring AI가 제공하는 **LLM 호출 통합 클라이언트**입니다.
- `.prompt().user(...).call().content()` 체이닝만으로 LLM에 질문하고 응답을 받을 수 있습니다.
- 내부적으로 Ollama HTTP API를 호출하지만, 코드에서는 그 세부사항이 보이지 않습니다. → **추상화의 힘**

### 3-2. `ChatClient.Builder` + `defaultAdvisors()` — Advisor 패턴

```java
builder.defaultAdvisors(profanityAroundAdvisor, new SimpleLoggerAdvisor())
```

- `defaultAdvisors()`에 등록한 Advisor는 **모든 LLM 호출 전·후에 자동 실행**됩니다.
- Spring AOP와 비슷한 개념입니다. `chatClient.prompt()...call()`을 호출하는 순간, 등록된 Advisor가 가로채서 파이프라인을 실행합니다.
- `SimpleLoggerAdvisor`는 Spring AI 내장 Advisor로, 요청/응답을 자동으로 로깅해줍니다.

### 3-3. `CallAroundAdvisor` — 핵심! LLM 호출을 가로채는 인터페이스

```java
// ProfanityAroundAdvisor.java
public class ProfanityAroundAdvisor implements CallAroundAdvisor {

    @Override
    public AdvisedResponse aroundCall(AdvisedRequest request,
                                      CallAroundAdvisorChain chain) {
        PipelineResponse result = pipelineService.run(request.userText());

        if (result.isProfanity()) {
            return buildBlockedResponse(request, result); // LLM 호출 없이 즉시 차단
        }
        return chain.nextAroundCall(request); // 안전하면 LLM 호출 진행
    }
}
```

- `CallAroundAdvisor`는 LLM 호출을 **감싸는(Around)** 인터페이스입니다.
- `chain.nextAroundCall(request)`를 호출해야만 LLM이 실제로 실행됩니다.
- 비속어가 탐지되면 `chain`을 호출하지 않고 직접 응답을 만들어 반환합니다. → **LLM 비용 절감 + 안전성 보장**
- `getOrder()` 값이 낮을수록 먼저 실행됩니다. 여기서는 `0`으로 최우선 실행을 보장합니다.

### 3-4. `VectorStore` — 벡터 DB 추상화

```java
// RagFilterService.java
vectorStore.similaritySearch(
    SearchRequest.builder()
        .query(normalized)
        .topK(props.rag().topK())
        .similarityThreshold(props.rag().similarityThreshold())
        .build()
);
```

- `VectorStore`는 Spring AI의 벡터 저장소 **공통 인터페이스**입니다.
- 이 코드에서는 pgvector(PostgreSQL 확장)를 구현체로 사용하지만, 코드 레벨에서는 `VectorStore`만 알고 있습니다.
- `similaritySearch()`에 쿼리 문자열을 넣으면, Spring AI가 **내부적으로 임베딩 모델을 호출**해 벡터로 변환한 뒤 유사도 검색을 수행합니다. 임베딩 변환 코드를 직접 쓰지 않아도 됩니다.

### 3-5. `Document` — Spring AI의 텍스트 + 메타데이터 단위

```java
// IngestService.java
new Document(word, Map.of(
    "category", request.category(),
    "source",   "ADMIN_DIRECT"
))
```

- `Document`는 Spring AI에서 텍스트와 메타데이터를 묶는 기본 단위입니다.
- `vectorStore.add(docs)` 시 Spring AI가 자동으로 임베딩 모델을 호출해 벡터로 변환 후 저장합니다.
- **메타데이터(Map)가 JSONB로 저장**되는 이유는 뒤 DB 설계 섹션에서 자세히 설명합니다.

## 4. 임베딩 모델: 왜 `bge-m3`를 선택했나요?

```yaml
# application.yaml
ai:
  ollama:
    embedding:
      options:
        model: bge-m3
```

임베딩 모델은 텍스트를 **숫자 벡터**로 변환하는 모델입니다.  
비속어 유사도 검색의 정확도는 임베딩 모델 품질에 직결됩니다.

**`bge-m3`를 선택한 이유:**

| 항목 | 설명 |
|------|------|
| 다국어 지원 | 한국어 포함 100개 언어 지원. 한국어 욕설·신조어도 잘 임베딩 |
| 벡터 차원 | **1024차원** — 표현력이 높아 유사한 욕설 패턴을 잘 구분 |
| 로컬 실행 | Ollama로 로컬에서 돌림 → API 비용 없음, 개인정보 외부 유출 없음 |
| BAAI 공개 모델 | 상업적 이용 가능한 오픈소스 |

```yaml
vectorstore:
  pgvector:
    dimensions: 1024  # bge-m3 출력 차원과 반드시 일치시켜야 함
```

> ⚠️ `dimensions`를 임베딩 모델 출력 차원과 맞추지 않으면 저장 시 에러가 납니다.

## 5. DB 설계

### 5-1. 테이블 3개 구조

#### `bad_word` — 비속어 블랙리스트 사전

```sql
CREATE TABLE bad_word (
    id         BIGSERIAL    PRIMARY KEY,
    word       VARCHAR(200) NOT NULL UNIQUE,  -- 욕설 원문 (UNIQUE 제약)
    category   VARCHAR(50)  NOT NULL DEFAULT 'ABUSE',
    source     VARCHAR(200),                  -- 출처 파일명
    is_active  BOOLEAN      NOT NULL DEFAULT TRUE,
    created_at TIMESTAMPTZ  NOT NULL DEFAULT now()
);
```

**설계 의도:**
- `word`에 `UNIQUE` 제약을 둔 이유: 같은 단어가 여러 번 적재되는 것을 DB 레벨에서 방지합니다. 애플리케이션 코드에서 중복 체크를 해도 동시 요청 시 레이스 컨디션이 생길 수 있기 때문에 DB 제약이 최후의 보루입니다.
- `is_active`를 둔 이유: 단어를 DELETE하면 이력이 사라집니다. `is_active = FALSE`로 비활성화하면 **언제 제외됐는지 추적**이 가능하고, 실수로 삭제해도 복구할 수 있습니다. 예를 들어 "구리다"가 욕설과 혼용될 수 있어 추가했다가 나중에 제외할 때 유용합니다.

**인덱스:**
```sql
-- JPA ddl-auto: update 가 자동 생성
-- UNIQUE 제약 → 자동으로 B-Tree 인덱스 생성
-- word 컬럼이 곧 인덱스 대상
```
- `word` 컬럼에 UNIQUE 제약을 걸면 PostgreSQL은 **자동으로 B-Tree 인덱스**를 생성합니다.
- `BadWordFilter`가 블랙리스트 매칭 시 DB를 조회하지 않고 메모리 `HashSet`을 사용하기 때문에, 이 인덱스는 주로 **중복 삽입 방지 체크**(`existsByWord`)에 활용됩니다.

---

#### `profanity_vectors` — 임베딩 벡터 저장소

```sql
CREATE TABLE public.profanity_vectors (
    id          UUID    DEFAULT uuid_generate_v4() NOT NULL,
    content     TEXT    NOT NULL,
    embedding   vector  NULL,                          -- pgvector 타입 (차원은 저장 시 자동 결정)
    metadata    JSONB   DEFAULT '{}'::jsonb NOT NULL,  -- Document 메타데이터 원본
    -- ▼ Generated Column: metadata JSONB에서 자동 추출
    category    TEXT    GENERATED ALWAYS AS (metadata ->> 'category')   STORED,
    severity    SMALLINT GENERATED ALWAYS AS (COALESCE((metadata ->> 'severity')::smallint, 1)) STORED,
    source_type TEXT    GENERATED ALWAYS AS (metadata ->> 'source_type') STORED,
    word_type   TEXT    GENERATED ALWAYS AS (metadata ->> 'word_type')   STORED,
    created_at  TIMESTAMPTZ DEFAULT now() NOT NULL,
    CONSTRAINT profanity_vectors_pkey PRIMARY KEY (id)
);
```

**설계 의도 — 항목별 설명:**

**① `id`를 UUID로 쓰는 이유**  
Spring AI `PgVectorStore`의 기본 규약이 UUID입니다. BIGSERIAL(순번)은 단일 DB에서는 단순하지만, 나중에 샤딩·복제 환경으로 확장하면 ID 충돌이 생길 수 있습니다. UUID는 **어느 노드에서 생성해도 충돌하지 않습니다.**

**② `embedding vector NULL` — 차원을 타입에 명시하지 않은 이유**  
`VECTOR(1024)`처럼 차원을 고정하면 임베딩 모델을 교체할 때(예: bge-m3 → bge-large, 차원 변경) **테이블 재생성이 필요**합니다. `vector NULL`로 두면 저장되는 첫 벡터의 차원이 자동으로 결정되어 모델 교체 시 유연성이 높아집니다. `application.yaml`의 `dimensions: 1024`는 Spring AI가 검색 쿼리 시 사용하는 설정입니다.

**③ `metadata`를 JSONB로 쓰는 이유**  
Spring AI `Document`의 메타데이터는 `Map<String, Object>` 형태입니다. 어떤 키가 들어올지 스키마 설계 시점에 고정하기 어렵고, 나중에 `tag`, `region` 같은 필드가 추가되더라도 **컬럼 ALTER 없이 확장**할 수 있습니다.  
JSON이 아닌 **JSONB(Binary JSON)** 를 쓰는 이유는 저장 시 파싱된 바이너리 형태로 저장하므로 조회 시 재파싱이 필요 없어 **읽기 성능이 빠르고**, GIN 인덱스를 걸어 **키 레벨 검색**이 가능합니다.

**④ Generated Column — 이 설계의 핵심**

```sql
category TEXT GENERATED ALWAYS AS (metadata ->> 'category') STORED
```

이것이 이 스키마의 가장 독특한 부분입니다.

- `metadata JSONB` 하나에 모든 메타데이터를 저장하되, 자주 필터링에 쓰이는 `category`, `severity` 등을 **별도 컬럼으로 자동 추출**합니다.
- `GENERATED ALWAYS AS ... STORED`: INSERT/UPDATE 시 PostgreSQL이 자동으로 계산해 저장합니다. 애플리케이션 코드에서 별도로 채울 필요가 없습니다.
- **이점**: `metadata`는 유연하게 확장 가능하면서, `category = 'ABUSE'` 같은 조건 검색은 B-Tree 인덱스로 빠르게 수행합니다. JSONB `->>`연산자 검색보다 훨씬 빠릅니다.
- **단점 없음**: `metadata`를 수정하면 Generated Column도 자동으로 갱신됩니다.

```java
// IngestService.java - metadata에 값을 넣으면 Generated Column이 자동 추출
new Document(word, Map.of(
    "category",    request.category(),  // → category 컬럼에 자동 반영
    "source_type", "ADMIN_DIRECT"       // → source_type 컬럼에 자동 반영
))
```

**인덱스 설계 — 5가지 인덱스와 선택 이유**

```sql
-- ① 벡터 유사도 검색용 (핵심)
CREATE INDEX idx_pv_embedding_hnsw
    ON profanity_vectors USING hnsw (embedding vector_cosine_ops)
    WITH (m='16', ef_construction='64');

-- ② JSONB 전체 키 검색용
CREATE INDEX idx_pv_metadata_gin ON profanity_vectors USING gin (metadata);

-- ③④⑤ Generated Column B-Tree 인덱스
CREATE INDEX idx_pv_category    ON profanity_vectors USING btree (category);
CREATE INDEX idx_pv_severity    ON profanity_vectors USING btree (severity);
CREATE INDEX idx_pv_source_type ON profanity_vectors USING btree (source_type);

-- ⑥ 최신순 정렬용
CREATE INDEX idx_pv_created_at  ON profanity_vectors USING btree (created_at DESC);
```

**① HNSW 인덱스 (가장 중요)**

pgvector는 두 가지 인덱스 알고리즘을 제공합니다:

| 인덱스 | 특징 | 적합 상황 |
|--------|------|-----------|
| IVFFlat | 벡터 공간을 클러스터로 나눠 근사 검색. 구축 빠름. 정확도 낮음 | 대용량, 배치성 데이터 |
| **HNSW** | 계층형 그래프로 탐색. 구축 느림. **검색 속도·정확도 모두 높음** | **실시간 서비스** |

비속어 필터는 사용자 입력마다 **실시간**으로 검색해야 합니다. HNSW가 검색 지연시간(latency)에서 압도적으로 유리하기 때문에 선택했습니다.

- `vector_cosine_ops`: **코사인 유사도** 기준으로 인덱스를 구성합니다. 텍스트 임베딩은 벡터의 방향 유사성이 핵심입니다. 두 벡터의 크기가 달라도 의미적으로 비슷하면 코사인 유사도가 높게 나옵니다. 반면 L2(유클리드 거리)는 벡터 크기 차이에 민감하므로 텍스트 유사도에는 코사인이 더 적합합니다.
- `m=16`: 각 노드가 연결하는 이웃 수입니다. 클수록 정확도↑ 메모리↑. 16은 pgvector 권장 기본값입니다.
- `ef_construction=64`: 인덱스 구축 시 탐색 깊이입니다. 클수록 인덱스 품질↑ 구축 시간↑. 64는 균형 잡힌 값입니다.

**② GIN 인덱스 on metadata**

```sql
CREATE INDEX idx_pv_metadata_gin ON profanity_vectors USING gin (metadata);
```

GIN(Generalized Inverted Index)은 JSONB의 **모든 키/값을 역인덱스**로 저장합니다. `metadata @> '{"category":"ABUSE"}'` 같은 포함 연산자(`@>`) 검색에 최적화됩니다. Generated Column 인덱스가 있어도 GIN이 있으면 복합 조건 검색 플래너가 더 효율적으로 동작합니다.

**③④⑤ Generated Column B-Tree 인덱스**

Generated Column이 있어도 인덱스가 없으면 전체 테이블 스캔(Seq Scan)을 합니다. `category`, `severity`, `source_type`에 B-Tree 인덱스를 걸면 `WHERE category = 'ABUSE'` 같은 필터가 **O(log N)** 으로 빠르게 실행됩니다. 관리자 화면에서 카테고리·위험도별 필터링 시 활용됩니다.

**⑥ `created_at DESC` 인덱스**

```sql
CREATE INDEX idx_pv_created_at ON profanity_vectors USING btree (created_at DESC);
```

`ORDER BY created_at DESC`(최신순) 쿼리에 최적화하기 위해 **내림차순으로 인덱스를 구성**합니다. 인덱스를 오름차순으로 구성하면 최신순 정렬 시 인덱스를 역방향으로 스캔해야 하므로 약간 비효율적입니다.

#### `llm_judge_log` — LLM 판정 기록 + 지속학습 루프

```sql
CREATE TABLE llm_judge_log (
    id              BIGSERIAL   PRIMARY KEY,
    original_text   TEXT        NOT NULL,
    normalized_text TEXT,
    llm_result      VARCHAR(20) NOT NULL,  -- PROFANITY | SAFE
    llm_reason      TEXT,
    confidence      FLOAT8,
    category        VARCHAR(50),
    severity        INTEGER,
    is_trained      BOOLEAN     NOT NULL DEFAULT FALSE,  -- 핵심 컬럼!
    reviewed_by     VARCHAR(100),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ
);
```

**설계 의도:**

이 테이블이 이 서비스의 **지속 학습(Continuous Learning) 루프**의 출발점입니다.

```
L3(LLM 판정) → llm_judge_log 저장 (is_trained=FALSE)
    ↓
관리자가 GET /api/admin/judge-logs 로 검토
    ↓
적절한 판정이면 POST /api/admin/ingest 로 벡터 DB에 등록 (is_trained=TRUE)
    ↓
다음 L2(RAG) 검색부터 이 패턴이 즉시 반영됨
```

- `is_trained = FALSE`: 아직 벡터 DB에 학습되지 않은 상태. 관리자 검토 대기.
- `is_trained = TRUE`: 관리자가 승인하여 `profanity_vectors`에 임베딩 완료.
- 이 구조 덕분에 **서비스가 운영되면서 스스로 강해집니다.** L3가 새로운 패턴을 잡으면 → 관리자 승인 → L2에 편입 → 다음번엔 L2가 먼저 차단.

**`updated_at`에 `@UpdateTimestamp` 사용 이유:**
```java
@UpdateTimestamp
private OffsetDateTime updatedAt;
```
Hibernate가 엔티티 update 시 **자동으로 현재 시각**을 채워줍니다. `is_trained`가 TRUE로 바뀐 시각을 자동 기록하여 "언제 승인됐는지" 추적합니다.

**`createdAt`, `updatedAt` 모두 `OffsetDateTime` 사용 이유:** `LocalDateTime`은 타임존 정보가 없어서 서버 타임존 설정에 따라 값이 달라질 수 있습니다. `OffsetDateTime`(= `TIMESTAMPTZ`)은 UTC 오프셋을 포함하므로 **어느 환경에서나 일관된 시각**을 보장합니다.

## 6. 필터 파이프라인 상세 설명

### 6-1. `filter-pipeline.yaml` — 왜 따로 분리했나요?

```yaml
# application.yaml 에서 import
spring:
  config:
    import: classpath:filter-pipeline.yaml
```

`application.yaml`에는 **인프라 설정**(DB URL, Ollama URL 등)이 있습니다.  
비즈니스 튜닝 값(정규식 패턴, 임계값 등)을 같은 파일에 넣으면 두 가지 문제가 생깁니다:

1. **담당자 혼선**: 인프라 담당자와 필터 정책 담당자가 같은 파일을 편집해야 함
2. **변경 이력 추적 어려움**: git blame 시 인프라 변경과 정책 변경이 뒤섞임

또한 `.properties` 대신 `.yaml`을 쓰는 데에도 이유가 있습니다:
- `.properties`는 ISO-8859-1 인코딩 기반으로 한글이 깨집니다.
- `.yaml`은 UTF-8이 기본이라 한글 정규식 패턴(`[시씨씪슈...]`)을 안전하게 저장할 수 있습니다.

### 6-2. `FilterPipelineProperties` — YAML 값을 타입 안전하게 바인딩

```java
@Validated
@ConfigurationProperties(prefix = "filter.pipeline")
public record FilterPipelineProperties(
    @Valid @NotNull Normalize normalize,
    @Valid @NotNull Regex regex,
    @Valid @NotNull Fuzzy fuzzy,
    @Valid @NotNull Rag rag
) { ... }
```

- `@ConfigurationProperties`는 YAML의 `filter.pipeline.*` 하위 값을 **타입이 있는 Java 객체**로 자동 바인딩합니다. `@Value("${filter.pipeline.fuzzy.threshold}")`를 여러 클래스에 반복 쓰지 않아도 됩니다.
- `@Validated` + `@NotNull`, `@DecimalMin`을 붙이면 앱 **시작 시** 설정값 유효성을 검사합니다. threshold에 `1.5`처럼 범위를 벗어난 값이 들어오면 바로 에러로 알 수 있습니다.
- `record`를 사용한 이유: Java 16+의 record는 불변(immutable) 객체입니다. 설정값은 런타임 중 바뀌면 안 되기 때문에 record가 적합합니다.

### 6-3. Step 1 — `TextNormalizer` (전처리)

```java
// TextNormalizer.java
public String normalize(String input) {
    // 1. 공백·특수문자 전면 제거 ("시 발" → "시발")
    String text = input.replaceAll("[\\p{Z}\\p{C}\\s]+", "");
    text = CLEAN_PATTERN.matcher(text).replaceAll("");

    // 2. 시각적 유사 기호 치환 ("7ㅣ발" → "ㄱㅣ발")
    text = applyVisualReplacements(text);

    // 3. 영타 → 한글 자모 변환 ("tlqkf" → "ㅅㅣㅂㅏㄹ")
    text = qwertyToJamo(text);

    // 4. NFC 정규화 + 반복 글자 압축 ("ㅋㅋㅋㅋ" → "ㅋㅋ")
    text = Normalizer.normalize(text, Normalizer.Form.NFC);
    return compressRepeat(text, props.normalize().maxRepeat());
}
```

**왜 이 순서인가요?**
- 공백·특수문자 제거를 가장 먼저 해야 이후 치환 패턴이 정확하게 적용됩니다.
- 시각적 치환 후 영타 변환을 하는 이유: `r`이 `ㅏ`로 치환되고 나면 Qwerty 매핑과 겹치지 않기 때문입니다.
- `compressRepeat`은 반복 회피("씨이이이이발")를 막습니다. `maxRepeat: 2`이면 같은 글자 연속 2개 초과분을 제거합니다.

**치환 규칙은 YAML에서 관리:**
```yaml
visual-replacements:
  "7": "ㄱ"
  "r": "ㅏ"
  "1": "ㅣ"
```
치환 규칙을 코드가 아닌 YAML에 두면, 새로운 우회 패턴이 발견될 때 **코드 수정 없이 YAML만 수정하고 재배포**할 수 있습니다.

### 6-4. Step 2 — Regex 매칭

```yaml
regex:
  pattern: '[시씨씪슈...][바발벌빠...]|...'
```

정규식은 **복합 패턴**을 한 번에 검사합니다.  
예를 들어 `[시씨][바발빠빡]`은 "시발", "씨발", "씨빨" 등 변형을 동시에 잡습니다.

패턴을 YAML에 두는 이유는 `TextNormalizer`와 동일합니다 — 코드 수정 없이 패턴 추가/제거 가능.

> 💡 `.properties` 파일이었다면 이 긴 한글 정규식이 `\uC2DC\uC36C...` 유니코드 이스케이프로 저장돼야 했을 것입니다. YAML 덕분에 그대로 쓸 수 있습니다.

### 6-5. Step 3 — Blacklist 1:1 매칭

```java
// BadWordFilter.java
public boolean matchesBlacklist(String text) {
    for (String token : text.split("\\s+")) {
        if (blacklistCache.contains(normalizeForCache(token))) return true;
    }
    return false;
}
```

- `bad_word` 테이블에서 활성 단어를 앱 시작 시 메모리 `HashSet`에 로드합니다.
- `HashSet`의 `contains()`는 **O(1)** 시간복잡도입니다. DB 조회 없이 매우 빠르게 검사합니다.
- 입력 텍스트를 `normalizeForCache()`로 동일한 방식으로 정규화해야 캐시와 비교가 정확합니다.

```java
// 앱 시작 시 1회 로드
@PostConstruct
public void initCache() {
    List<String> words = badWordRepository.findAllActiveWords();
    this.blacklistCache = new HashSet<>(words.stream()
        .map(this::normalizeForCache).toList());
    this.nfdCache = words.stream().map(this::toNfd).toList();
}
```

DB를 **1회만 조회**해서 `blacklistCache`(Regex/Blacklist용)와 `nfdCache`(Fuzzy용)를 **동시에 구성**합니다. 요청마다 DB를 조회하면 성능이 크게 나빠지기 때문입니다.

### 6-6. Step 4 — Fuzzy NFD N-Gram 매칭

가장 복잡한 단계입니다. "씨이이이이이발"처럼 정규화 후에도 완전 일치하지 않는 변형을 잡기 위해 사용합니다.

**라이브러리: `java-string-similarity`의 `Jaccard`**

```xml
<dependency>
    <groupId>info.debatty</groupId>
    <artifactId>java-string-similarity</artifactId>
</dependency>
```

이 라이브러리를 선택한 이유:
- **Jaccard N-Gram 유사도**를 한 줄로 계산할 수 있습니다.
- Apache Commons Text의 Levenshtein은 편집 거리 기반이라 글자 수 차이에 민감합니다. Jaccard는 **공통 N-Gram 비율** 기반이라 글자가 삽입/삭제되어도 유사도가 안정적입니다.

**NFD 자모 분해를 쓰는 이유:**
```java
private String toNfd(String text) {
    return Normalizer.normalize(text, Normalizer.Form.NFD);
}
```
한글 "발"은 NFD로 분해하면 `ㅂ + ㅏ + ㄹ` 3개의 자모가 됩니다.  
"빨"은 `ㅂ + ㅏ + ㄹ` + 쌍자음 변형 = `ㅃ + ㅏ + ㄹ`  
자모 레벨에서 비교하면 두 글자의 유사도가 높게 나옵니다. **문자 레벨보다 더 세밀한 유사도 계산**이 가능합니다.

**슬라이딩 윈도우 검사:**
```java
for (int w = Math.max(1, bLen - 1); w <= Math.min(inputNfd.length(), bLen + 1); w++) {
    for (int i = 0; i <= inputNfd.length() - w; i++) {
        String slice = inputNfd.substring(i, i + w);
        double score = jaccard.similarity(slice, badNfd);
        if (score >= threshold) return true;
    }
}
```
입력 텍스트가 길 때, 욕설 단어 길이 ±1 크기의 창으로 **잘라가며 비교**합니다. "나는 씨발 나쁜 놈이야" 같이 욕설이 문장 중간에 있어도 탐지됩니다.

**임계값 `threshold: 0.75`를 선택한 이유:**

```yaml
fuzzy:
  threshold: 0.75
```

| threshold | 효과 |
|-----------|------|
| 1.0 | 완전 일치만 허용 → Fuzzy 의미 없음 |
| 0.5 이하 | 너무 관대 → 정상 단어도 탐지(오탐 폭증) |
| **0.75** | 한 글자 정도 변형 탐지 가능 + 오탐 최소화 (실험적 권장값) |

이 값은 정답이 없습니다. 오탐/미탐 비율을 직접 테스트해가며 조정해야 합니다. YAML에 있기 때문에 **코드 변경 없이 튜닝**할 수 있습니다.

### 6-7. L2 — `RagFilterService` (RAG 유사도 검색)

```java
// RagFilterService.java
List<Document> docs = vectorStore.similaritySearch(
    SearchRequest.builder()
        .query(normalized)
        .topK(props.rag().topK())               // 5
        .similarityThreshold(props.rag().similarityThreshold()) // 0.82
        .build()
);
boolean isProfanity = !docs.isEmpty();
```

**`topK: 5`를 선택한 이유:**
- `topK`는 유사도 상위 몇 개 문서를 가져올지 설정합니다.
- 1개만 가져오면 우연히 유사한 정상 문서가 1위일 때 오탐이 발생할 수 있습니다.
- 5개를 가져왔을 때 하나라도 비속어 벡터가 걸리면 탐지로 판정합니다.
- 너무 많이 가져오면(예: 50개) 검색 범위가 넓어져 오탐이 증가합니다. 5는 **정확도와 오탐률의 균형점**입니다.

**`similarityThreshold: 0.82`를 선택한 이유:**
- 0.0~1.0 사이에서 1.0에 가까울수록 **완전히 일치하는 문서만** 반환합니다.
- 0.82는 "의미적으로 비슷한" 수준입니다. 완전 같은 단어가 아니라도 욕설 계열 텍스트가 검색됩니다.
- 너무 낮으면(0.5 이하) 전혀 관계없는 문서도 걸려 오탐이 많아집니다.
- 이 값도 YAML에서 관리하므로 운영 중 조정이 가능합니다.

**RAG(Retrieval-Augmented Generation)이란?**  
원래 RAG는 "검색된 문서를 LLM 프롬프트에 포함시키는 패턴"을 말합니다.  
이 프로젝트에서는 LLM 호출 없이 **검색 결과 유무 자체**를 비속어 판단에 사용합니다. RAG의 Retrieval 부분만 활용한 변형 패턴입니다.

### 6-8. L3 — `LlmFilterService` (LLM 최종 판정)

L1, L2를 모두 통과한 **모호한 케이스**만 여기까지 옵니다.

```java
// LlmFilterService.java
String llmRaw = llmChatClient.prompt()
    .system(sp -> sp.text(systemPromptResource).param("context", "없음"))
    .user(normalized)
    .call()
    .content();
```

**`@Qualifier("llmChatClient")`를 쓰는 이유 (순환참조 방지):**

```
chatClient(Primary)
  └─ ProfanityAroundAdvisor
       └─ PipelineService
            └─ LlmFilterService
                 └─ chatClient(Primary) ← 여기서 순환!
```

`chatClient`에 `ProfanityAroundAdvisor`가 붙어있기 때문에, `LlmFilterService`가 같은 `chatClient`를 쓰면 무한 루프가 발생합니다. 그래서 **Advisor 없는 전용 빈** `llmChatClient`를 별도로 만들어 주입합니다.

```java
// ChatClientConfig.java
@Bean @Qualifier("llmChatClient")
public ChatClient llmChatClient(ChatClient.Builder builder) {
    return builder.build(); // Advisor 없음
}
```

### 6-9. `profanity-judge-system.st` — 프롬프트 설계

```
# prompts/profanity-judge-system.st
당신은 한국어 언어 안전 검사 시스템입니다.
...
비속어인 경우: RESULT:PROFANITY
정상인 경우:   RESULT:SAFE
...
[유사 비속어 목록]
{context}
```

**`.st` 파일을 별도로 분리한 이유:**
- 프롬프트를 코드에 하드코딩하면 수정할 때마다 재배포해야 합니다.
- `classpath:prompts/profanity-judge-system.st`로 분리하면 프롬프트만 수정 후 재배포할 수 있습니다.
- Spring AI의 `Resource` 타입으로 주입받으면 자동으로 파일을 읽어줍니다.

**출력 형식을 `RESULT:PROFANITY` 고정 포맷으로 강제한 이유:**
- LLM은 자유 형식으로 답하면 "저는 이 문장이 비속어라고 생각합니다..." 같은 불필요한 서술을 붙입니다.
- 고정 포맷을 강제하면 **정규식으로 파싱**이 가능합니다.

```java
// LlmFilterService.java - LLM 응답 파서
private static final Pattern P_RESULT = Pattern.compile("RESULT:(PROFANITY|SAFE)");
private static final Pattern P_REASON = Pattern.compile("REASON:(.*?)(?=CATEGORY:|SEVERITY:|$)", Pattern.DOTALL);

private static boolean parseIsProfanity(String raw) {
    var m = P_RESULT.matcher(raw);
    return m.find() && "PROFANITY".equals(m.group(1));
}
```

LLM 응답을 직접 boolean으로 쓸 수 없기 때문에, **정규식 파서 4개**(`parseIsProfanity`, `parseReason`, `parseCategory`, `parseSeverity`)로 각 필드를 추출합니다. 파서를 `private static`으로 구현해서 클래스 내부에서만 사용하도록 캡슐화했습니다.

**`temperature: 0.1`로 낮게 설정한 이유:**
```yaml
chat:
  options:
    temperature: 0.1
```
temperature는 LLM 응답의 창의성(무작위성) 수치입니다. 비속어 판단은 창의적일 필요가 없고, **일관성과 재현성**이 중요합니다. 낮을수록 같은 입력에 같은 출력을 냅니다.

## 7. `ProfanityAroundAdvisor` 상세 흐름 (Spring AI 핵심)

```
POST /api/chat
    │
    ▼

    │  chatClient.prompt().user(text).call().content()
    ▼
[Spring AI ChatClient 내부]
```

**`AdvisedResponse` 구조:**
```java
return new AdvisedResponse(chatResponse, Map.of(
    "profanity_blocked", true,
    "detected_by",       result.detectedBy(),
    "pipeline_ms",       result.pipelineMs()
));
```
두 번째 인자 `Map`은 Advisor 컨텍스트 데이터입니다. 다음 Advisor나 후처리 로직에서 이 값을 꺼내 사용할 수 있습니다.

## 8. API 호출 순서와 클래스 흐름 (시연 순서)

### 시연 1: L1 단독 테스트

```http
POST /api/filter/l1-local
{"text": "tlqkf"}
```

```
FilterController.l1LocalFilter()
  └─ LocalFilterService.checkAsMap()
       └─ TextNormalizer.normalize()       → "tlqkf" → "ㅅㅣㅂㅏㄹ"
       └─ BadWordFilter.matchesRegex()     → 패턴 매칭
       └─ BadWordFilter.matchesBlacklist() → HashSet 조회
       └─ BadWordFilter.matchesFuzzy()     → Jaccard N-Gram
```

### 시연 2: L2 단독 테스트

```http
POST /api/filter/l2-rag
{"text": "시발"}
```

```
FilterController.l2RagFilter()
  └─ RagFilterService.checkAsMap()
       └─ LocalFilterService.normalize()        → 전처리
       └─ VectorStore.similaritySearch()        → bge-m3 임베딩 후 pgvector 검색
            └─ SearchRequest(topK=5, threshold=0.82)
```

### 시연 3: L2(RAG) + L3(LLM) 통합 테스트

```http
POST /api/filter/l3-llm
{"text": "너 지능이 좀 아쉽네"}
```

```
FilterController.l3LlmFilter()
  │
  ├─ [Step 1] RagFilterService.check()
  │       └─ LocalFilterService.normalize()      → 전처리
  │       └─ VectorStore.similaritySearch()      → pgvector 유사도 검색
  │            └─ 탐지 시 → 즉시 차단 응답 반환 (LLM 호출 없음)
  │
  └─ [Step 2] RAG 미탐지 시 LlmFilterService.check()
          └─ llmChatClient.prompt().system(st파일).user(text).call()
               └─ Ollama(llama3) 호출
          └─ parseIsProfanity() / parseReason() / ...
          └─ LlmJudgeLogRepository.save()  → llm_judge_log 저장
```

**응답 예시 (RAG 미탐지 → LLM 탐지):**
```json
{
  "originalText": "너 지능이 좀 아쉽네",
  "isProfanity": true,
  "detectedBy": "L3-LLM",
  "ragMatched": [],
  "llmResponse": "RESULT:PROFANITY\nREASON:지능을 비하하는 표현\nCATEGORY:ABUSE\nSEVERITY:2",
  "message": "LLM 최종 판정에서 비속어로 탐지되어 차단되었습니다."
}
```

**순화 출력 대신 차단을 선택한 이유:**
- LLM이 "씨발 → 이런"으로 바꿔도 맥락상 여전히 공격적일 수 있어 순화 결과를 100% 신뢰할 수 없습니다.
- 이 서비스의 역할은 "필터"이지 "편집기"가 아닙니다. 책임이 섞이면 설계가 복잡해집니다.
- 차단해야 `llm_judge_log`에 원문이 쌓여 **지속학습 루프**(log → 관리자 승인 → 벡터 DB 편입)가 정상 동작합니다. 순화하면 원문이 통과되어 로그가 의미 없어집니다.

### 시연 4: 통합 파이프라인 (Advisor 동작)

```http
POST /api/chat
{"text": "씨이이이이발"}
```

```
FilterController.chat()
  └─ chatClient.prompt().user(text).call()
       └─ [ProfanityAroundAdvisor 진입]
            └─ PipelineService.run()
                 └─ L1: TextNormalizer → "씨이이이이발" → "씨이발"(압축)
                 └─ L1: BadWordFilter.matchesFuzzy() → 탐지!
            └─ buildBlockedResponse() ← LLM 호출 없음
       └─ [ProfanityAroundAdvisor 반환]
```

### 시연 5: 관리자 → 지속 학습

```http
# 1단계: 판정되지 않은 로그 조회
GET /api/admin/judge-logs

# 2단계: 해당 로그를 벡터 DB에 등록
POST /api/admin/ingest
{"logIds": [1, 2, 3], "category": "PROFANITY"}
```

```
AdminController.ingest()
  └─ IngestService.ingest()
       └─ LlmJudgeLogRepository.findAllById()  → 로그 조회
       └─ refinedLog.setIsTrained(true)        → 학습 완료 마킹
       └─ new Document(normalizedText, metadata) → Spring AI Document 생성
       └─ VectorStore.add(docs)               → bge-m3 임베딩 후 pgvector 저장
```

## 9. 로그로 보는 파이프라인 흐름

코드 곳곳에 SLF4J `log.info()`를 넣어 단계별 통과/탐지를 추적할 수 있습니다.

```
[L1-Step1] 원문='씨@발' → 정규화='씨발'
[L1-Step2] 정규식(Regex) 검사: 탐지
[Pipeline-STOP] L1 로컬 필터에서 차단됨
```

```
[L1-Step1] 원문='너 지능이 좀 아쉽네' → 정규화='너지능이좀아쉽네'
[L1-Step2] 정규식(Regex) 검사: 통과
[L1-Step3] 1:1 매칭 검사: 통과
[L1-Step4] Fuzzy NFD 매칭 검사: 통과
[L2-RAG] isProfanity=false matched=0 | '너 지능이 좀 아쉽네'
[L3-LLM] 판정: true | 카테고리: ABUSE | 위험도: 2
[Pipeline-FINISH] 최종 판단: PROFANITY (소요시간: 1240ms)
```

로그만 봐도 어느 단계에서 차단됐는지, 각 단계를 어떻게 통과했는지 한눈에 파악됩니다.  
`logging.level.com.lily.spring_ai_vector: DEBUG`로 설정해두면 더 상세한 Fuzzy 매칭 점수도 볼 수 있습니다.

## 10. 전체 정리: 이 프로젝트에서 배운 것

| Spring AI 개념 | 이 프로젝트에서의 활용 |
|----------------|----------------------|
| `ChatClient` | LLM 호출 추상화 (Ollama llama3) |
| `ChatClient.Builder` + `defaultAdvisors` | 파이프라인 자동 실행 설정 |
| `CallAroundAdvisor` | LLM 호출 전·후 비속어 필터링 가로채기 |
| `VectorStore` | pgvector 임베딩 저장·검색 추상화 |
| `Document` | 텍스트 + 메타데이터 단위 (JSONB로 저장) |
| `SearchRequest` | topK, similarityThreshold 기반 유사도 검색 |
| `SimpleLoggerAdvisor` | 요청/응답 자동 로깅 Advisor |

> **핵심**: Spring AI의 추상화 덕분에 Ollama를 OpenAI로 바꾸거나, pgvector를 Pinecone으로 바꾸더라도 **이 코드는 거의 수정하지 않아도 됩니다.** 이것이 Spring AI를 쓰는 가장 큰 이유입니다.
